### Problem Statement (Plain English)
MCP gives agents a connection layer for tools, but security decisions are split across clients, servers, prompts, approval dialogs, and logs — nobody has defined what execution-layer security looks like. This paper asks whether we can make execution-layer security invariants explicit and testable in a runtime, while keeping MCP-like workflows intact.

### Methodology (Technical)
The paper defines **eight security invariants** for MCP-style agent execution: (1) metadata non-authority, (2) grant-backed approval, (3) canonical resources, (4) principal binding, (5) scoped capability invocation, (6) source-and-target data-flow authorization, (7) deny-path audit, and (8) explicit protocol state. They implement these in **HCP** (Handle-Capability Protocol), a reference runtime for MCP-style execution that models calls through principals, resources, grants, capabilities, handles, policy decisions, data-pipe checks, and audit entries. They evaluate HCP against: (a) a naive connection-layer runtime, and (b) a practice-informed mitigation baseline (metadata linting + session checks + per-call approvals). The benchmark covers 10 attack cases. An ablation study identifies which runtime components block which attacks.

### Key Results
- **Naive baseline:** permits all 10 modeled attacks
- **Mitigation baseline:** permits 6 of 10 attacks
- **HCP:** blocks all 10 attacks while preserving audit evidence
- **Latency:** sub-millisecond mean for measured policy, invocation, peek, and pipe operations
- Ablations show which components provide which defensive coverage

### What's Novel
First formal treatment of execution-layer security invariants for MCP-style agent systems. The HCP reference implementation proves these invariants are implementable with negligible latency overhead. Prior work focused on connection-layer security (auth, transport) or model-level attacks (jailbreaks) — this paper fills the execution-control gap.

### My Connection
Directly applicable to mcpsec's runtime security model. The eight invariants provide a formal specification for what mcpsec should enforce. HCP's principal-resource-grant-capability pattern could inform mcpsec's sandboxing architecture. The benchmark of 10 attack cases is a ready-made test suite for validating mcpsec's execution controls.

### What I Learned (Plain English)
Connection-layer security (who can connect) is not enough — you need execution-layer security (what the agent can do once connected). The HCP paper shows that with the right design, you can enforce eight strong security invariants with sub-millisecond overhead. Every agent runtime should implement at least the principal binding and scoped capability invocation invariants.
